# ADR 007: DDDドメイン境界の設計方針

## ステータス

採択済み (2026-04-27)

## 背景

本プロジェクトは Clean Architecture + DDD（ドメイン駆動設計）を採用しており、`backend/domain/` 配下に集約・エンティティ・値オブジェクト・ドメインサービス・リポジトリインターフェースを配置している。

しかしこれまで、「なぜこの単位で境界を引いたのか」「どのルールで依存方向を決定したのか」を明示したドキュメントが存在しなかった。本ADRはその設計判断を記録する。

## 問題

DDDを適用する際、以下の判断が曖昧なまま実装が進むと、後から矛盾した依存関係や不整合な集約設計が混入するリスクがある。

1. どの単位で集約を切るか（集約の大きさ・小ささのトレードオフ）
2. エンティティと値オブジェクトの区別基準
3. 認証に関するオブジェクト（RefreshToken、WebAuthnCredential）をどのドメインに属させるか
4. ドメインサービスを使うべきかエンティティメソッドに入れるべきか
5. Application層のPortsパターンと外部サービス依存の分離

## 決定

### 1. 集約の境界

**採択**: `FinancialPlan` を中心集約とし、`User`・`RefreshToken`・`WebAuthnCredential` は別の集約として独立させる。

```
集約1: FinancialPlan（集約ルート）
  └─ FinancialProfile（エンティティ）
  └─ Goal（エンティティ × 複数）
  └─ RetirementData（エンティティ）
  └─ EmergencyFundConfig（値オブジェクト）

集約2: User（集約ルート）

集約3: RefreshToken（エンティティ）
集約4: PasswordResetToken（エンティティ）
集約5: WebAuthnCredential（エンティティ）
```

**理由**:
- `FinancialProfile`・`Goal`・`RetirementData` は財務計画の一貫性（Consistency）を一緒に保証する必要があるため、同一集約に含める
- `User` は認証・識別の責務を持ち、財務計画と独立したライフサイクルを持つ（ユーザー削除で財務計画も削除されるが、逆は成立しない）
- `RefreshToken`・`WebAuthnCredential` はトークンの有効期限管理・セッション管理が主な責務であり、`User` と同一集約にすると集約が肥大化してロード/更新のコストが上がるため分離した

**却下した設計**:
- `User` を集約ルートとして `FinancialPlan` を内包する案: 単一集約が巨大になりすぎ、トランザクション境界も不明確になる
- `Goal` を独立集約にする案: `Goal.IsAchievable()` が `FinancialProfile` の情報を必要とするため、同一集約内にある方が整合性の保証が容易

### 2. エンティティと値オブジェクトの区別基準

**採択**: 同一性（Identity）が必要かどうかで判断する。

| 分類 | 基準 | 例 |
|---|---|---|
| **値オブジェクト** | 同一性が不要。値が同じなら同じものとみなせる | `Money`、`Rate`、`Period`、`Age`、`Email`、`GoalType` |
| **エンティティ** | ID によって同一性を識別する必要がある | `User`、`Goal`、`FinancialProfile`、`RetirementData` |

**理由**:
- `Money(10000, JPY)` は金額と通貨が同じであれば、どのオブジェクトも等価とみなせるため値オブジェクト
- `Goal` は同じ目標額・目標日であっても「どの目標か」を ID で識別する必要があるためエンティティ
- 値オブジェクトを不変（immutable）にすることで、意図しない副作用を防げる（Goでは値型で渡すことで実現）

### 3. ドメインサービスの使用基準

**採択**: 複数の集約/エンティティにまたがるビジネスロジック、または単一エンティティに属するには不自然なロジックをドメインサービスに置く。

| サービス | 理由 |
|---|---|
| `FinancialCalculationService` | 複利計算・インフレ調整は `Money`・`Rate` を入力とする純粋な計算。特定エンティティの状態に依存しないため |
| `GoalRecommendationService` | 推奨額の算出は `FinancialProfile` と `Goal` の両方を参照する。単一エンティティのメソッドに置くと依存が逆転する |

**却下した設計**:
- `FinancialProfile.CalculateCompoundInterest()` として実装する案: 計算ロジックが特定エンティティに結合し、再利用性が下がる。テストも複雑になる

### 4. Ports & Adapters パターンの適用範囲

**採択**: Application Layer に `ports/` パッケージを設け、外部サービスのインターフェースをここで定義する。

```
application/ports/
  llm_client.go    # LLM（Claude等）への問い合わせインターフェース
  faq_loader.go    # FAQ データの読み込みインターフェース
```

**理由**:
- `BotUseCase` が LLM API に直接依存すると、テスト時に実際の API 呼び出しが発生する
- ポートとして抽象化することで、テスト時はモック実装、本番時は実装を DI できる
- LLM プロバイダの変更（例: Claude → OpenAI）がアプリケーション層に影響しない

**適用しなかった箇所**:
- メール送信（`AuthUseCase` 内で直接 Resend HTTP API を呼ぶ）: 現時点でプロバイダ変更の可能性が低く、抽象化のコストが利益を上回ると判断

### 5. 集約間の結合ルール

**採択**: 異なる集約をまたぐ参照は、オブジェクト参照ではなく ID（`UserID` 型）を用いる。

```go
// 良い例: IDで参照
type Goal struct {
    id     GoalID
    userID UserID  // User集約への参照はIDのみ
    ...
}

// 避けるべき例: オブジェクト参照
type Goal struct {
    user *User  // 集約をまたいだ直接参照は禁止
    ...
}
```

**理由**:
- オブジェクト参照を許すと、一方の集約のロードが他方を引き込み、トランザクション境界が崩れる
- ID 参照にすることで、集約ごとに独立したリポジトリから取得でき、キャッシュ戦略も集約単位で適用できる（→ ADR-006）

## 重視するアーキテクチャ特性

- **変更容易性（Modifiability）**: 集約境界を明確にすることで、変更の影響範囲を集約内に限定できる
- **テスト容易性（Testability）**: ドメインサービスをエンティティと分離し、インターフェース経由でリポジトリに依存することで、単体テストが書きやすい
- **保守性（Maintainability）**: 設計判断をADRとして記録し、将来の開発者が同じ議論を繰り返さないようにする

## 代替案と却下理由

**Transaction Script パターン**: ユースケースに手続き的に全ロジックを書く。シンプルだが、ビジネスルールが分散し大規模になると保守が困難。

**大きな集約（God Aggregate）**: `User` に全データを内包する。整合性確保は容易だが、ロード/保存のコストが高く、同時編集での競合が多発する。

**Anemic Domain Model**: エンティティをデータ構造のみにして、ビジネスロジックをサービス層に集約する。Goと相性が悪くはないが、ビジネスルールがドメインモデルから流出しテストが書きにくくなる。

## 影響

- `backend/domain/` の境界は本ADRの判断基準に従って維持する
- 新しいエンティティ/値オブジェクトを追加する際は、本ADRを参照して分類を判断する
- 集約をまたぐ整合性が必要な操作は、Application Layer のユースケース内で複数リポジトリを調整して実現する（ドメインイベントは現時点では未採用）
- 詳細な依存関係図は [DOMAIN_DIAGRAM.md](../architecture/DOMAIN_DIAGRAM.md) を参照
