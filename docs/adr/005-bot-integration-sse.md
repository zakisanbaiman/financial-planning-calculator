# ADR 005: ボット応答の返却方式として SSE を採用

## ステータス

提案中 (2026-03-31)

## 背景

Issue #217「ボット追加」では、アプリケーションに対話型のボット機能を追加したい意図が示されている。ただし、現時点のIssue本文は「sseでメッセージ返却するかADR作成から」とだけ記載されており、実装着手前に返却方式の決定が必要な状態である。

本プロジェクトは Next.js フロントエンドと Go/Echo バックエンドで構成されており、既存APIは主にREST/JSONベースである。一方、ボット機能では以下の要件が想定される。

- 主目的は FAQ 回答である
- 利用者はログインユーザーに限定する
- 応答は1文字ずつ、シームレスに表示したい
- 双方向リアルタイム通信は不要である
- 会話履歴は初期段階では保存しない
- 初期実装ではローカル LLM を利用する
- FAQの知識源は Markdown を正本として管理したい
- 長時間処理でも待機中の体験を悪化させたくない
- 実装・運用コストはできるだけ抑えたい

## 問題

ボットの応答返却方式を決めないまま実装を始めると、フロントエンドのUI設計、バックエンドのAPI設計、タイムアウト制御、監視方法が後からずれやすい。

特に以下の選択肢に差がある。

- 通常の同期HTTPレスポンスで一括返却する
- SSE (Server-Sent Events) で逐次返却する
- WebSocket で双方向通信する

## 決定

**ボットのメッセージ返却方式として、初期実装では SSE (Server-Sent Events) を採用する。**

具体的には、ボット応答を開始した後、バックエンドは1本のHTTP接続上でイベントを逐次送信し、フロントエンドは `EventSource` または `fetch` ストリーム経由で受信してUIへ反映する。

初期スコープは以下とする。

- FAQ回答を主目的としたボットを対象とする
- クライアントからボットへの送信は通常のHTTPリクエストで行う
- ボットからクライアントへの応答のみをストリーミング対象とする
- 1文字ずつに近い粒度でメッセージを逐次表示できるようにする
- まずは「1リクエストに対して1応答ストリーム」の単純なモデルを採用する
- 会話履歴の永続化や複数ルーム対応は別Issueで扱う
- 利用対象はログインユーザーに限定し、既存JWT認証と整合する構成を前提にする
- FAQの知識源は `docs/faq/` などの Markdown 群を正本として管理する
- 初期のモデル実行基盤はローカル LLM とする

## この決定で重視するアーキテクチャ特性

この決定は、特に以下のアーキテクチャ特性を優先している。

1. **変更容易性（Modifiability）**
   - 初期実装を小さく始め、後から WebSocket や非同期ジョブ基盤へ拡張しやすくする
   - 既存の REST/HTTP ベース設計を大きく壊さずに導入できる

2. **保守性（Maintainability）**
   - 接続管理や状態同期の複雑さを抑え、実装者が追いやすい構成にする
   - フロントエンドとバックエンドの責務を比較的単純に保てる

3. **ユーザビリティ / 応答性（Usability / Responsiveness）**
   - FAQ応答を逐次表示することで、待機時間の体感を改善する
   - 長い応答でも「動いている」ことを利用者に伝えやすい

4. **運用性（Operability）**
   - 既存のHTTP監視・ログ収集の延長で扱いやすい
   - WebSocket専用の接続監視や状態管理を初期段階では避けられる

5. **実装コスト効率（Cost Efficiency）**
   - PoC段階で必要以上に複雑な基盤を持たず、学習コストと開発コストを抑える
   - ローカル LLM を前提とした小さな検証を進めやすくする
   - 価値検証を早め、必要に応じて後続投資できる形にする

6. **知識の管理容易性（Knowledge Maintainability）**
   - FAQ本文を Markdown で管理し、非エンジニアでも差分を追いやすくする
   - LLMの知識源をコードから分離し、更新作業を小さく保つ

## アーキテクチャ特性のトレードオフ

- **変更容易性・保守性を優先**する代わりに、WebSocketほどの双方向リアルタイム性は初期段階では持たない
- **ユーザビリティを改善**できる一方で、通常の同期APIより実装とテストはやや複雑になる
- **運用性は比較的高い**が、長時間接続やプロキシのバッファリングなど、SSE特有の注意点は残る

## 理由

### SSEを選ぶ理由

1. **実装が比較的単純**
   - 既存のHTTP基盤（Go/Echo、ブラウザ、プロキシ）に乗せやすい
   - WebSocketのような接続管理・再接続設計を最初から重く持たなくてよい
   - これは主に **保守性** と **実装コスト効率** を満たす

2. **ボット応答の性質に合う**
   - 初期段階では「クライアントから1回送信し、サーバーが段階的に返す」片方向ストリーミングで十分
   - FAQ用途かつ双方向不要という要件に合っている
   - LLMのトークン逐次表示や進捗イベント送信と相性が良い
   - これは主に **ユーザビリティ / 応答性** を満たす

3. **ユーザー体験が向上する**
   - 応答生成中でも文字や進捗を順次描画でき、待ち時間の心理的負荷を下げられる
   - 1文字ずつシームレスに表示したいという要件に適合する
   - タイムアウトや失敗時も途中経過をUIで扱いやすい
   - これは主に **ユーザビリティ / 応答性** を満たす

4. **段階的導入に向いている**
   - まずは最小構成で導入し、必要になったら WebSocket やジョブキューへ発展させやすい
   - API仕様を大きく壊さずにPoCから本実装へ移行しやすい
   - これは主に **変更容易性** と **保守性** を満たす

5. **Markdown FAQ と相性が良い**
   - FAQ文書を小さく保つ限り、検索基盤なしでも実用的な応答を返しやすい
   - 将来、Markdown を正本のまま検索用インデックスだけ追加しやすい
   - これは主に **知識の管理容易性** と **変更容易性** を満たす

### WebSocketを初期採用しない理由

- 双方向通信、接続維持、再接続、セッション管理などの考慮点が増える
- 現時点の要件ではオーバースペックになりやすい
- インフラや監視の複雑性が先に立ち、Issue #217 の前進が遅くなる

### 一括JSONレスポンスを採用しない理由

- 応答完了までUIに変化が出しにくい
- 長文応答や外部AI API待ちが入ると体感速度が悪い
- 後からストリーミング化するとAPI/UI双方の変更範囲が大きくなる

## 想定アーキテクチャ

### バックエンド

- `POST /api/bot/messages` などでユーザー入力を受け付ける
- レスポンスは `text/event-stream` を返却する
- イベント種別は最低限 `message`, `done`, `error` を持つ
- プロキシ環境でバッファリングされないようヘッダー制御を行う
- 既存JWT認証と連携し、未ログインユーザーは利用不可とする
- FAQの正本は `docs/faq/` 配下の Markdown とし、起動時または更新時に読み込めるようにする
- ローカル LLM を呼び出すアダプタ層を設け、将来的なクラウドLLM切り替え余地は残す
- FAQ 件数が少ない初期段階では、質問に関連しそうな Markdown をそのままプロンプトへ注入する単純方式を採用する

### フロントエンド

- ボット送信後、受信イベントを逐次表示するチャットUIを持つ
- `done` イベントで応答完了扱いにする
- `error` イベントまたは接続断でリトライ導線を表示する
- FAQ用途を前提に、自由会話よりも「質問して回答を得る」UIを優先する

### 運用・監視

- リクエスト数、平均応答時間、ストリーム切断数を観測対象にする
- 長時間接続に備えてタイムアウトとキャンセルを明示的に扱う
- 認証が必要な場合は既存JWT認証との整合を保つ
- ローカル LLM の応答時間、失敗率、メモリ使用量を観測できるようにする

## 代替案

1. **同期HTTP + JSON**
   - 利点: もっともシンプル
   - 欠点: 体験が弱く、将来的な拡張時に再設計コストが高い

2. **WebSocket**
   - 利点: 双方向・リアルタイム性に優れる
   - 欠点: 現時点では要件過多で、初期導入コストが高い

3. **ポーリング**
   - 利点: 実装しやすい
   - 欠点: レイテンシ、無駄な通信、状態管理の複雑化が起きやすい

## 結果

### 期待される効果

- Issue #217 の次アクションを「SSE前提のAPI/UI実装」に絞り込める
- ボットの応答体験を早い段階で検証できる
- 将来的なAI連携やストリーミング出力に備えた基盤を先に作れる
- FAQ の更新を Markdown 編集ベースで進められる
- ランニングコストを抑えつつローカル環境で検証を始められる

### 今後の課題

- ローカル LLM の具体的なモデル選定
- 会話履歴を将来保存する必要が出た場合の拡張方法
- SSEを利用する具体的なAPI仕様とテスト戦略の策定
- Markdown FAQ の更新フローとレビュー手順の整備

## FAQ知識源の段階的拡張方針

FAQデータ量や応答品質の要件が大きくなった場合でも、Markdown を正本として残したまま次の順で拡張する。

1. **初期段階**
   - `docs/faq/` の Markdown をそのまま利用する
   - カテゴリや見出しベースで対象FAQを絞ってプロンプトへ渡す

2. **FAQ増加時**
   - Markdown から正規化JSONを生成する
   - タイトル、タグ、見出し、本文を構造化し、単純なキーワード検索を追加する

3. **さらに重くなった時**
   - Markdown/JSON を元に埋め込みを生成してベクトル検索を追加する
   - RAG構成に移行し、関連チャンクのみをローカル LLM へ渡す

4. **運用が本格化した時**
   - FAQ管理UIやCMS導入を検討する
   - ただし正本が Markdown から完全に分離されるかは別ADRで判断する

この方針により、初期の **保守性** と **コスト効率** を保ちながら、将来の **拡張性** と **性能** にも対応しやすくする。
